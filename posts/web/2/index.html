렌더링 과정 (크롬기준)

렌더링 과정

html문서를 받아온 시점부터 브라우저 content로 화면에 그려지는 부분까지를 정리 해봅니다.

브라우저 구조

일단 브라우저마다 구조는 조금씩 다르지만 기본적인 구조를 보면

  • User Interface: 주소 표시줄, 이전/다음 버튼, 북마크 메뉴 등. 요청한 페이지를 보여주는 창을 제외한 나머지 모든 부분
  • Browser Engine: User Interface와 Rendering Engine 사이의 동작을 제어
  • Rendering Engine: 요청한 콘텐츠를 표시, HTML을 요청하면 HTML과 CSS를 파싱 하여 화면에 표시함
  • Networking: HTTP 요청과 같은 네트워크 호출에 사용됨
  • Javascript Interpreter(또는 Engine): 자바스크립트 코드를 해석하고 실행함. 크롬에서는 V8 엔진을 사용함
  • Display Backend: 기본적인 위젯(콤보 박스 등..)을 그림
  • Data Persistence: Local Storage, 쿠키 등 클라이언트 사이드에서 데이터를 저장하는 영역

이제 위의 구조 중 렌더링 엔진에대해 정리 해보려합니다.

렌더링 엔진

렌더링 엔진에는 content가 pixel로 그려지기 까지의 파이프라인이 있습니다.

parse <-> style <-> layout <-> paint <-> change

Parse

DOM tree를 생성합니다.

network 요청이든 자신의 파일이든 html과 같은 content를 객체로 파싱합니다. html같은 경우는 단순한 텍스트로 입력되고 렌더링 엔진은 이를 조작할 수 있는 객체로 변환하는데, 이 과정을 파싱 이라고 합니다.

Blink는 크로미움의 렌더링 엔진입니다. Blink에서는 HTMLDocumentParser 로 텍스트를 읽으면서 HTMLTreeBuilder 로 Document Object Model (DOM) 트리를 하나씩 완성합니다. 이렇게 단순한 텍스트였던 문서는 JS로 콘텐츠를 조작할수 있는 모델이 되었습니다.

Style

스타일링 단계에서는 CSS 텍스트 파일에 정의된 스타일을 DOM 에 적용할 수 있도록 해석하는 작업을 진행합니다.

CSSParser를 통해 스타일 규칙, 스타일 셀렉터에대한 정보 정리 및 속성과 속성 값들을 객체화 합니다. 약간 DOM을 만드는것처럼 CSS에대한 CSSOM을 만드는 과정입니다. 여기서 만들어진 객체들을 DOM과 연결하여 사용할 준비가 되었습니다.

Layout

DOM도 만들어졌고 styleSheetContents 또한 만들어 졌으면 다음단계로 Layout 작업이 시작됩니다.

레이아웃 단계는 각 요소들이 화면에서 어디에 배치할지를 결정하는 작업입니다.

기본적인 전체 레이아웃 구성관련한 트리를 만들고 이후 뷰포트에 맞춰 작용하는 속성들에대한 업데이트 또한 필요합니다.

Paint

레이아웃 작업을 통해 만들어진 화면상의 위치정보 또한 만들어 졌습니다. 이제 텍스트, 색, 이미지, 그림자 효과와 같은 부분들이 처리되어 그려집니다.

먼저 화면에 무슨 타입의 효과와 색 그라데이션 등이 그려질지 페인팅 작업들을 보관하는 객체를 만듭니다. 이후 래스터라이징이라는 과정을 통해 실제 이미지를 만드는 단계를 거칩니다.

래스터라이징 작업은 그래픽 연산이기 때문에 그래픽처리장치(GPU) 의 도움을 받아 빠르게 이미지를 만들도록 되어 있습니다.

Change

paint단계까지 마치면서 어렵게 화면을 구성하는데 성공했습니다. 이후 스크롤이나 사용자에 의해 화면이 변경될 수 있는데, 이 때문에 처음부터 모든걸 다시 재구성할 수 없습니다.

화면은 1초당 60개의 프래임을 보여줍니다. 자바스크립트 실행과정이나 페이지 스크롤, 이미지 등 위에서 본 여러 구간에서의 과정이 길어져 렌더링을 제대로 이루지 못하면 page junk가 발생합니다.

그래서 이러한 변화에대해 대응하는 방법들을 알아보겠습니다.

유효성검사

화면 재구성에서 단계(style, layout, paint, raster)별로 유효성검사를 통해 업데이트가 필요한 부분의 데이터만 변경하는 방법입니다.

compositing

레이어 합성입니다. 영상처리 기술에서는 이미 많이 알려진 방법이라고 하는데, 큰 화면을 레이어로 쪼개서 작업하고 관리하면 화면의 일부인 애니메이션을 표현하기 위해 전체 이미지를 계속 다시 그릴필요가 사라집니다.

tiling

화면을 거의 차지하는 스크롤 레이어가 있다면 합성의 효과도 미미해집니다. 레이어를 더 작은 단위인 타일로 쪼개고 타일의 위치나 특성에 따라 우선순위를 매겨 각 타일별 이미지를 만듭니다. 이렇게 하면 현재화면(viewport)에 아직 보여지지 않는 화면보다 보여지고 있는 화면의 우선순위를 둬 해당 부분을 더 빠르게 렌더링합니다.

activation

힙겹게 만들어진 레이어 이미지는 먼저 ‘pending tree’ 에 commit 되고, 다음 ‘draw’ 이벤트 전에 레이어를 합성하여 레스터라이징합니다.

레스터라이징이 끝나서 실제 화면에 표현할 준비가 되면 ractivation 작업을 통해 ‘pedding tree’ 를 ‘active tree’ 에 반영합니다. ‘active tree’ 는 특정 간격마다 draw 작업을 진행해서 렌더링된 이미지를 화면에 뿌립니다.
