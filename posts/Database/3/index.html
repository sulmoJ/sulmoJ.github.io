데이터베이스 정규화에 대해서


Entity, Relation, table관계

개념 모델링 시 추출한 개체를 엔터티라 부르고, 논리 모델링을 통해 물리 모델링으로 엔터티를 테이블로 구성하고 테이블을 관계형 데이터 모델로 정의할 때는 릴레이션R이라고 합니다.

어떻게 보면 셋 다 같은 듯 보이지만, 보통은 개념/논리에서 도출된 엔터티(Entity)를 변환해서 물리의 테이블을 만듭니다. 하지만, 성능 등을 고려해 정규화 또는 비정규화를 통해 물리적인 구성은 변환할 수 있습니다. 그래서 엔터티 그대로 테이블이 되지 않는 경우도 있습니다. 예를 들면 슈퍼/서브타입의 경우가 있답니다.

정규화는 왜 생겼나?

한 릴레이션에 여러 엔티티의 애트리뷰트들이 혼합시 일부 속성은 중복저장이 되어 저장공간 낭비가 되기도하고

여러 릴레이션에 중복으로 존재하는 속성들 때문에 한쪽에서 변경된 같은 속성의 정보가 다른 릴레이션에 동기화가 안되있을 수도 있다.

이러한 갱신 이상이 발생한다. 해당 문제를 해결하기 위해 정규화 과정을 진행한다.

갱신이상 종류

  • 삽입 이상 : 자료 삽입에 있어 자료가 부족해 삽입할수 없는 문제, 또는 우너하지 않는 자료 삽입
  • 삭제 이상 : 하나의 자료만 삭제하려 했으나 그 자료가 포함된 다른 속성전체가 삭제되는 손실 우려
  • 수정 이상 : 일부의 튜플만 갱신되어 정보가 모호해져 일관성이 없어지는 이상

정규화란?

관계형 DB에서 중복을 최소화하기 위해 데이터를 구조화 하는 작업

함수적 종속관계를 파악해 정규화의 종류가 나눠진다.

일단 함수적 종속성에대한 개념을 이해 하자.

일단 정의는 X의 값을 알면 Y는 바로 식별 가능하고 X값에 Y값이 달라짐.

예) 학번, 나이, 이름 속성이 있을 때, 나이,이름은 학번에 함수적 종속이다. 학번 -> 이름로 표현하기도 한다.

이러한 함수적 종속은

  • 완전 함수 종속
  • 부분 함수 종속
  • 이행적 함수 종속

이있다. 아래 정규형과 함께 설명 하겠습니다.

제 1 정규형

테이블에서 컬럼이 원자값 을 갖도록 테이블을 분해하는 것

한 컬럼에 값이 두개가 들어가있는 튜플을 나눠 제 1 정규형을 만족 시킨다.

예)

이름 좋아하는 고기
설모 소, 양
술모 닭,오리

에서 |이름|좋아하는 고기| |---|---| |설모|소| |설모|양| |술모|닭| |술모|오리|

제 2 정규형

제 1 정규화를 진행한 테이블에 대해 완전 함수 종속을 만족하도록 테이블 분해

기본키의 부분 집합이 결정자가 되어선 안된다

* 결정자 : A->B에서 A

학번 강좌이름 강의실 성적
1 테트리스 101 4.5
2 테트리스 101 4.3
2 드로잉 102 2.3

를

학번 강좌이름 성적
1 테트리스 4.5
2 테트리스 4.3
2 드로잉 2.3
강좌이름 강의실
테트리스 101
드로잉 102

이렇게 나눕니다.

제 3 정규화

제 2 정규화를 진행한 테이블에 이행적 종속을 없애도록 테이블을 분해

학생번호 강좌이름
501 데이터베이스
401 데이터베이스
402 스포츠경영학
502 자료구조
강좌이름 수강료
데이터베이스 20000
스포츠경영학 15000
자료구조 25000

BCNF 정규화

제 3 정규화를 진행 후 테이블에 대해 모든 결정자가 후보키가 되도록 테이블을 분해하는 것

정규화 장단점

  • 장점

    • 갱신 이상의 문제점들 해결
    • 데이터베이스 구조 확장시 기존 구조를 수정해야하는 일이 적다. 확장에 용이하다
    • 사용자에게 데이터 모델을 더욱 의미있게 제공

  • 단점

    • 릴레이션이 많이 분해되어있어 릴레이션간 JOIN연산이 늘어난다. 추가적인 작업을 통해 가공됨으로 응답 시간이 느려질 수 있고

단점에대한 대처

단점을 위해 다시 정규화를 일부 위반한다. 반정규화라 부르며 서비스와 구조에 따라 조회가 많거나 특수한 경우에대한 테이블을 정규화를 무시해서라도 합친다.
