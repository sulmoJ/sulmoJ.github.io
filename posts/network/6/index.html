HTTP와 HTTPS의 차이, 관계


HTTP

HTML문서와 같은 리소스들을 가져올 수 있도록 해주는 애플리케이션 계층 프로토콜

애플리케이션 계층의 프로토콜인 HTTP는 TCP/IP위에서 작동하고 상태를 가지고 있지 않은 Stateless 프로토콜이다. 상태를 가지고 있지 않다는건 이전 통신과의 독립적으로 아무 관계가 없는 새로운 요청이다.

그 이후 지속적인 연결을 위해 쿠키, 세션의 개념이 등장 함

HTTP의 문제점

  • 도청가능
  • 요청자가 위장가능
  • 데이터 변조가능
도청가능

TCP/IP는 도청 가능한 네트워크구조이다.

전송계층의 TCP/IP 구조의 통신은 통신 경로 상에서 엿볼 수 있다. 패킷을 통해 평문으로 된 정보들은 모두 파악이 가능해 암호화를 해야한다.

요청자가 위장가능

HTTP는 통신 상대가 누구인지 확인하는 처리가 없다. 그래서 누구든지 요청할 수 있고, IP 주소나 포트 등에서 그 웹서버단 액세스 제한이 없는 경우 누구든지 응답한다.

데이터 변조 가능

완전성(정보의 정확성)을 증명할 수 없어 송신측에서 보낸 정보와 일치하는지 알수 없다. 응답이 출발하고 수신하기전 프록시에서 데이터를 변조가능하다. 이를 중간자 공격이라고 부른다.

HTTPS

HTTP 에 암호화와 인증, 그리고 완전성 보호를 더한 HTTPS

HTTPS는 새로 만들어진 프로토콜이 아니다. 전송계층에 TCP와 응용계층에 HTTP 사이에 추가적인 계층과 함께 통신 자체를 암호화 하는 SSL이나 TLS라는 프로토콜을 추가해 암호화 하는 방식이다.

TCP <-> SSL <-> HTTP

이렇게 통신을 하며 암호화를 하고 SSL의 특징이 추가되어

  • 상대를 확인하는 수단으로 증명서를 제공
  • 제 3자 기관에 의해 발행되는 것이기 때문에 서버나 클라이언트가 실재하는 사실을 증명
  • 해당 증명서로 웹 사이트 인증에서도 활용

  • 공개키/개인키 암호화 방식을 이용한다.

    • 서버측이 인증된 기관으로부터 개인키를 받고 인증기관은 공개키를 브라우저에 기록해 서버측으로부터 암호화된 정보를 받을 수 있게 한다.

상황별 정리

  • HTTP 사용 : 암호화의 과정이 없기 때문에 CPU리소스를 덜먹음으로 중요 정보를 다루지 않는 일반적 정보 배포는 HTTP를 사용하는 것이 좋다.
  • HTTPS : 개인정보를 필요로하는 서비스관련 페이지는 HTTPS를 사용해 보안적 문제를 해결해 사용한다.

하지만 최근 하드웨어의 발달로 HTTPS를 사용해 하드웨어가 힘들어한다던지..는 없기때문에 왠만하면 HTTPS를 사용하는 방향으로 바뀌어 가는 중이다.
